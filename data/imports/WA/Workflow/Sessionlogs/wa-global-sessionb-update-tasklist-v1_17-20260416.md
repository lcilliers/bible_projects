## Design Differences from v5.0 Stage 2 — Recorded

These are the principal changes from v5.0 Stage 2 that the new instruction must implement:

**D1 — Session start from Stage 1 Completion Record, not from extract**
v5.0: Session B starts by confirming the extract version.
New: Analysis Output starts by reading the Stage 1 Completion Record. This provides: extract version (already confirmed in Stage 1); 7-domain checklist pass state; Path 3 notes count; SD pointer count already in database; catalogue questions indexed for this word. The extract version is not re-confirmed independently — it is taken from the Completion Record as authoritative.

**D2 — Stage 2a is one comprehensive document, not six structured passes**
v5.0: Six passes each with defined scope, reading list, type (b) writes, pass-close patch.
New: Stage 2a is a single free-form narrative session. Everything Claude AI observes about the word from reading the entire extract goes into one document. No structure is imposed. No database writes occur during Stage 2a. The document is then fixed — it does not change in Stage 2b or 2c. Stage 2a is complete when the document has been produced, downloaded, and confirmed.

**D3 — The catalogue is the organising principle for Stage 2b, not six thematic lenses**
v5.0: Six passes organised by analytical theme (meaning, divine, verses, somatic, language, correlation).
New: Stage 2b assigns catalogue questions to the observations from Stage 2a. Each Q&A pair consists of a catalogue question and an answer drawn from the Stage 2a narrative. The catalogue drives what is asked; Stage 2a provides the evidence; Stage 2b is the structured bridge between them. Database writes (Type (b) patches) occur at Stage 2b session close.

**D4 — Stage 2a covers all data that v5.0 spread across six passes**
Stage 2a must cover: all terms (OWNER and XREF); all verse context groups and descriptions; all anchor verses (with the five cross-registry vision questions active throughout); all lexical data; all correlation signals; all existing findings as input material; all thin-evidence phase2 flags from Stage 1; all existing SD pointers in the database; Path 3 notes from Stage 1. Nothing may be omitted from Stage 2a on the grounds that "it is covered in a later pass." There is no later pass.

**D5 — SD pointers are raised throughout Stage 2a, not accumulated for a Pass 6**
v5.0 integrity rule SB-13: Pass 6 must not be the only pass that generates SD pointers.
New: SD pointers are raised throughout Stage 2a as observations are recorded — per GR-OBS-001. They are recorded in the observations log and persisted to the database at Stage 2b close.

**D6 — Stage 2c replaces the Analytical Brief**
v5.0: Analytical Brief (nine sections, internal document) + Session C word study (separate Session C process).
New: Stage 2c produces the Analytic Word Output — the six Session C chapters. This is a reader-facing document, not an internal brief. It serves both the Session C and Session D handoff functions. The Session C process reads from Stage 2c, not from a brief.

**D7 — Characteristic-perspective grouping model governs Stage 2a reading**
v5.0: Pass 3 reads every anchor verse with five standing cross-registry questions.
New: Stage 2a reads every anchor verse with both the group description and the five standing cross-registry questions. The group description names what the verse is about in inner-being terms. The verse is read in that light. This is the characteristic-perspective grouping model (GR-PROG-006) applied at the reading level.

**D8 — Closure is a separate stage, not embedded in Stage 2**
v5.0: Stage 3 (closure) immediately follows Stage 2 passes within the same instruction.
New: Closure is a named stage after Stage 2c — it has its own checklist, corrective action loop, closing patch, and handoff signal. The structure mirrors the Analysis Readiness completion logic: verify, confirm, declare.

## Fellowship Stage 1 Run — Findings Relevant to Stage 2 Design

Registry 062 (fellowship) Stage 1 completed. Key findings relevant to Stage 2a design:

- **19 verse context groups** across two OWNER terms (koinōnia, koinōnos) — the full inner-being load is carried by Greek. Hebrew ח-ב-ר root has extensive physical/architectural corpus entirely set aside and documented for Session D.
- **Two analytical axes** already visible from group structure: divine-human correspondence (participation in God/Christ) and relational disposition (mutual bond among believers). Stage 2a must read these as they are — not impose a pass structure that would separate them artificially.
- **Path 3 notes** from Stage 1 are specific — `god_as_subject` and `somatic_link` verification items that Stage 2a must address when reading anchor verses.
- **Zero B-target flags** at Stage 1 close — clean handoff.
- **35 existing SD pointers** already in the database from prior work — Stage 2a reads these as context, not as completed analysis.

These findings confirm: Stage 2a for a Greek-dominant registry with two terms and 19 groups is a bounded reading task. The instruction must be specific about what comprehensive means — reading all 19 groups' anchor verses is the minimum floor, not a ceiling.

### T-2A — Draft Stage 2a: Comprehensive analysis → Observations log
**Status:** IN PROGRESS
**Output:** Stage 2a section of wa-sessionb-analysis-output-v1-20260416.md

**What Stage 2a must specify:**

**Opening — inputs and session start:**
- Required inputs: Stage 1 Completion Record; verified clean extract (version from Completion Record); master catalogue (for awareness of what Stage 2b will ask)
- Session start protocol: read Completion Record first; confirm extract currency; load Path 3 notes; note existing SD pointer count; confirm catalogue question count for this word
- Do not begin reading until all inputs confirmed

**Tracking documents:** Same four-section structure as Analysis Readiness but different content:
- SD Pointer Accumulator (replaces Patch Accumulator — SD pointers recorded on discovery throughout)
- RESEARCHER_DECISION Accumulator (same format)
- Path 3 Resolution Notes (records Stage 1 Path 3 items as they are addressed in anchor verse reading)
- Stage 2a Progress Record (position markers — what has been read and recorded)

**What to read and in what order:**
1. Stage 1 Completion Record (already done at session start)
2. Registry overview: word, cluster, sb_classification, dimensions, description
3. All XREF terms: owning registry, meaning — understand the borrowed vocabulary
4. All OWNER terms: meaning, root family, related words, LSJ (Greek), semantic range — before reading any verses
5. All verse context groups: group descriptions in full — understand the characteristic-perspective landscape before reading individual verses
6. Correlation signals: ranked pairs, shared anchor verses — understand the cross-registry context before reading
7. Existing SD pointers in the database — understand what prior analysis raised
8. Path 3 notes from Stage 1 — understand what needs verification
9. Existing findings as input material — understand what prior analysis found; these are not confirmed; they are starting hypotheses

Then anchor verse reading — per group, per term:
10. For each group: read all anchor verses. For each anchor verse: apply the five cross-registry vision questions and the group description as the lens. Record immediately — per GR-OBS-001.
11. For each set-aside verse: read the set-aside reason. Note if the reason is absent (from Stage 1 IC-06). Do not analyse set-aside verses — confirm they were correctly set aside.
12. Thin-evidence phase2 flags from Stage 1: read the verses for each thin-evidence term. Record disposition.

**What to record — free-form, nothing withheld:**
- Everything the data says about this word — meaning, structure, usage, patterns, anomalies
- Every SD pointer raised — immediately, per GR-OBS-001
- Every Path 3 note resolved — record what the verse evidence showed
- Every observation about prior findings — confirmed, deepened, questioned, or set aside
- No conclusion held back on grounds of uncertainty — uncertain observations are recorded as uncertain

**What Stage 2a explicitly does not do:**
- Does not produce structured findings for the database — that is Stage 2b
- Does not answer catalogue questions — that is Stage 2b
- Does not write reader-facing prose — that is Stage 2c
- Does not apply the Closure Checklist — that is after Stage 2c
- Does not construct patches — that is Stage 2b close

**Sign-off condition:**
Stage 2a is complete when:
(a) Every OWNER term's meaning and semantic range has been addressed
(b) Every group description has been read
(c) Every anchor verse across all groups has been read with the five cross-registry questions applied
(d) All thin-evidence phase2 flags are dispositioned
(e) All Path 3 notes are addressed
(f) All existing findings reviewed as input material
(g) All correlation signals reviewed
(h) All SD pointers recorded (minimum: all existing pointers reviewed; new ones raised as found)
(i) Observations log downloaded and confirmed

**Multi-session management:** Stage 2a may span multiple sessions. The Stage 2a Progress Record tracks which groups and terms have been read. Each session adds to the log. The log is fixed (no further writing permitted) once the Stage 2a sign-off is recorded.

### T-2B — Draft Stage 2b: Q&A partitioning → Q&A log + Type (b) patches
**Status:** PENDING — begins after T-2A confirmed
**Output:** Stage 2b section of wa-sessionb-analysis-output-v1-20260416.md

**What Stage 2b must specify:**

**Input:** Stage 2a observations log (fixed); catalogue questions indexed for this word.

**Process — three passes per session:**
- Pass 1: Read all catalogue questions for this word. For each question: identify whether the Stage 2a observations log contains evidence to answer it. Assign: answered / partially answered / not answered / not applicable.
- Pass 2: For each answered or partially answered question: write the Q&A pair. Question from catalogue. Answer drawn directly from Stage 2a — no new analysis. If the answer requires more than what Stage 2a contains: note the gap; Stage 2a must be re-opened (this is a Stage 2a gap, not a Stage 2b gap).
- Pass 3: For each not-answered question where the answer should exist: determine whether (a) Stage 2a was silent because the data genuinely does not answer it (mark not applicable), or (b) Stage 2a missed something (reopen Stage 2a for that specific item).

**Universal questions:** After word-specific questions are processed, work through the master catalogue questions (universal scope). Same three-pass approach.

**SD pointers from Stage 2a:** Review all SD pointers accumulated in Stage 2a. For each: confirm the pointer is correctly scoped and the description is precise. Add to Stage 2b patch for persistence.

**Type (b) patch at Stage 2b close:**
- `wa_session_b_findings`: one finding per Q&A pair — `finding_type`, `finding` (the answer text), `anchor_verses`, `pass_ref = 'Stage 2b'`, `term_id` where applicable
- `wa_finding_catalogue_links`: one link per finding–question pair
- `wa_session_research_flags`: SD pointers from Stage 2a — persisted as SD_POINTER records
- `evidential_status` updates on `wa_term_inventory` where Stage 2a reading determined classification

**Iterative:** Stage 2b may require multiple sessions. Each session picks up from the last completed question in the progress record.

**Sign-off condition:** All catalogue questions (word-specific and universal) have a disposition; Type (b) patch applied and confirmed; SD pointer count in database matches Stage 2a log count.
