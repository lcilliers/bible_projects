## The problem, deeper

M38 ran end-to-end in a single day but the wall clock was 12 h 31 min while the active processing was 6 h 45 min. The dead time — 5 h 46 min — was nearly half the day. The largest single dead-time block was the 4 h gap between Phase B close (10:03) and Phase D start (14:06), but the pattern repeated at smaller scale at every phase boundary: 5–10 minutes here and there as I posted a status, waited for you to see it, oriented you to where things were, took the next instruction.

Yesterday you said: *"I am very likely lost. It was difficult to follow the process as much is done in the background, and surfacing notes are often stale."* This is the root experience the gate-management refinement has to address.

There are three distinct things happening at every phase boundary that the current pipeline doesn't distinguish:

1. **Mechanical work** — the next phase's input file needs to be generated from the previous phase's validated output. There is no judgment here. A script can do it.
2. **Confirmation** — the previous phase's output should be visible to you, but you wouldn't change anything based on what you see. You're glancing, not deciding.
3. **Judgment** — the previous phase produced something where the next action depends on a real choice between substantively different paths. Only you can decide.

In M38, every phase boundary mixed all three. So you, returning from other commitments, had no way to know which kind of gate you were facing. The safe assumption was "judgment", which meant you slowed down on every gate to make sure you weren't missing something. The pipeline paid the cost in dead time.

The refinement is not to remove your involvement — it's to make the right kind of involvement available at the right gate.

## The model

### Three gate types and how to identify them

| Gate type | Test | Action |
|---|---|---|
| **Mechanical** | Can a script produce the next-phase input from the previous-phase validated output with no judgment? | CC executes. No alert. Researcher sees in dashboard. |
| **Confirmation** | Would the researcher want to glance at this before proceeding, but wouldn't typically change anything? | CC writes summary. Auto-advances unless researcher actively pauses. |
| **Judgment** | Is there a real choice between two or more substantively different paths, where the choice depends on knowledge only the researcher has? | CC alerts. Pauses. Waits for explicit response. |

A fourth class isn't a gate but matters: **failed validation**. When a hygiene check fails (truncation, gap, schema error), CC enters propose-validate-repair (Document 3). This is not a judgment gate; it's automated recovery. If recovery fails after the configured retry cap, then it becomes a judgment gate.

### Applied to v3_0 phases

For each phase boundary, what gate type applies:

| Boundary | Gate type | Why |
|---|---|---|
| Phase A start → UT review | Mechanical | Inputs come from DB; no decisions |
| UT review → researcher | Confirmation if no rescue candidates surfaced; **Judgment** if any | Bias-flag rescues need explicit decision |
| UT → Pass A | Mechanical | Input is just the IB-relevant verses |
| Pass A → keyword analytics | Mechanical | Input is the new meanings |
| Pass A → researcher | Confirmation | Researcher might glance; usually doesn't change anything |
| Phase A → Phase B.1 input | Mechanical | Input is just the meanings corpus structured for debate |
| B.1 → researcher | **Judgment** if BOUNDARYs returned; Confirmation if all STAYS | BOUNDARY handling is the canonical judgment decision |
| B.1 → B.2 input | Mechanical (after BOUNDARY decisions captured) | Input is the affirmed-to-include verse list |
| B.2 → researcher | Confirmation if hygiene checks pass; **Judgment** if distribution warnings | 40% ceiling warnings need explicit confirmation |
| B.2 → B.3 input | Mechanical | Input is per-sub-group verse list |
| B.3 → researcher | Confirmation if hygiene checks pass; **Judgment** if anchor warnings | Multi-verse anchor choice is judgment |
| Phase C apply | Mechanical | Deterministic SQL transaction |
| Phase D Stage A start | Mechanical | Catalogue prompts × characteristics already known |
| D.A → D.B input | Mechanical | Input is the char findings as evidence |
| D.B → researcher | Confirmation | Same shape as D.A; glance not decide |
| End of D → researcher | **Judgment** | Audit verdict may surface real issues; researcher decides |
| Phase E start | Mechanical (assumes audit cleared) | Inputs generated by existing script |
| Phase E.0 vision → ch1 | Confirmation | Glance the vision; doesn't typically change |
| Each chapter → next | Mechanical | Drafts assembled |
| Consistency pass → researcher | **Judgment** | Edit list may need researcher to weigh in |

The pattern: most boundaries are mechanical. The judgment gates are exactly where you'd want them — BOUNDARY decisions, audit-verdict review, consistency edits. Confirmation gates exist but auto-advance.

This means a well-running v3_0 pipeline for a clean cluster (no BOUNDARYs, no audit issues, no anchor warnings) could run end-to-end with **two or three judgment-gate pauses** rather than the ten or twelve in M38.

### B. The URGENT.md file

A single file at repo root that's mostly empty. When a judgment gate is hit, CC writes a focused brief at the top.

**Path:** `URGENT.md` at repo root.

**State when no gate is active:**

```markdown

_No pipeline gates require attention. Last checked: 2026-05-30 09:14:23_
```

**State when a judgment gate is active:**

### C. VS Code notifications — current state of knowledge

I'm honestly uncertain what the integration supports. CC running as a VS Code extension can do many things; whether triggering native OS notifications is one of them depends on what the integration exposes. The IDE-context section of my prompt mentions the VS Code integration but doesn't enumerate notification capabilities.

The pragmatic path: try the URGENT.md approach first (which is reliable). If after M36 it's clear that you still miss the URGENT signal because you don't have the IDE focused, investigate native notifications then.

A middle approach worth considering: at every judgment gate, CC can write a single bell character to stdout. If your terminal is open and has bell-on-alert enabled, you get an audible cue. Cheap; works only when the terminal is focused.

### D. Gate-type taxonomy in v3_0 instruction

v3_0 today doesn't name gate types. The refinement adds a §X.1 "Gate types" that:

1. Defines the three types with the test for each.
2. For each phase boundary, declares the gate type.
3. Names the actions CC takes for each type.

This becomes the contract between CC and you: when CC tells you a gate is mechanical, it means "I'm proceeding"; when it tells you a gate is confirmation, it means "I'm proceeding unless you say otherwise within a configurable window"; when it tells you a gate is judgment, it means "I'm waiting for you".

### E. Auto-advance through mechanical and confirmation gates

The implementation pattern: at the end of each phase's script, after hygiene checks pass, CC immediately:

1. Updates the dashboard.
2. Generates the next phase's input file.
3. If the next gate is mechanical, runs that phase's API call.
4. If the next gate is confirmation, posts a brief summary to the obslog and dashboard, and waits a configurable interval (e.g. 5 minutes) before auto-advancing.
5. If the next gate is judgment, writes URGENT.md and waits indefinitely.

The 5-minute auto-advance interval on confirmation gates is the key change: it lets you intervene if you want, but doesn't make you intervene if you don't.

## Open uncertainties

1. **VS Code notification capability** — I don't know for certain whether CC in the current VS Code integration can trigger native OS notifications. URGENT.md works regardless.

2. **The right confirmation window** — 5 minutes is a guess. Could be 1 minute or 30 minutes. The right answer depends on how distracted you typically are. Cluster-runtime would benefit from shorter windows; safety would benefit from longer.

3. **What counts as "judgment" can drift** — for example, B.1 BOUNDARYs might be judgment for the first cluster of a category and become routine (confirmation) by the third. The taxonomy may need per-cluster overrides as patterns emerge.

4. **Failed-validation handling vs judgment-gate handling** — when propose-validate-repair fails twice, is the resulting gate a judgment gate or a "this needs investigation" alert? The proposal treats it as judgment, but in practice you might want a different framing because the issue isn't a choice between paths — it's a script bug or an API hiccup.

## Questions for your review

1. **Does the three-gate-type model match your experience of M38?** Or are there gate types it misses? For example, is there a "shape review" gate that's neither pure confirmation (you do want to look) nor judgment (you wouldn't typically change anything) that needs its own treatment?

2. **The dashboard format above — is the section list right?** Specifically: are "Today's timeline", "Open issues", "Failures this run", "Cost ledger", "Next gates expected" the right five sections? Is anything missing? Anything you'd never use?

3. **URGENT.md as a separate file, or integrated into the dashboard?** Two files is more discoverable but redundant. One file is cleaner but easier to ignore if it's not screaming "look at me".

4. **The auto-advance interval on confirmation gates** — does 5 minutes feel right? 1 minute too aggressive; 30 minutes defeats the purpose?

5. **VS Code notifications** — worth investigating now, or defer until we see whether URGENT.md is enough?

6. **Per-cluster gate-type overrides** — do you want the ability to say "for this cluster, treat B.2 hygiene warnings as judgment, not confirmation"? Or trust the taxonomy?
