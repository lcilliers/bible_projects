The programme moves each word through a defined sequence of phases. Each phase has an input it receives, an output it produces, and an inflection point that must be satisfied before the word advances. The sequence is not a rolling conveyor; it is a disciplined progression in which every phase's output is persisted into the database, and the database state is what entitles a word to enter the next phase. A word that has not satisfied its current phase stays where it is; no phase is skipped; no assessment made at one phase is treated as final before the next phase confirms it.

The phases are six in number, named in the order in which they run.

**Session A — Term extraction.** For each registry word, Session A assembles the complete source data: the Hebrew and Greek terms that carry the word's meaning, the full set of verses in which those terms occur, and the lexicon data for every term. The source is STEP Bible, from which the verse inventory is retrieved and prepared in a structured form. Session A runs in two phases within a single working window — a data-preparation phase that produces a source file for researcher verification, and an import phase that produces the database import record. Its output is the complete per-word dataset: every term, every verse, every lexical attribute, ready to be read. Its inflection point is import into the database and confirmation that the data is complete for that word.

**Verse Context — Verse reading and grouping.** Once a word's data is in the database, the Verse Context stage reads every verse in which the word's terms occur. Each verse is filtered for relevance — does it engage the inner being through this term, or is the term present in a sense that falls outside the programme's question? Relevant verses are grouped by contextual meaning. Groups describe what a verse is about — the inner-being characteristic it engages — rather than what the term does within the sentence. The same term can therefore serve different characteristics across its corpus, and different groups can hold different angles on the same word. Each group is given an anchor verse — the citation that represents the group most clearly and that serves as the primary input to subsequent analysis. The output is a complete map of the word's verses: which are in, how they cluster, which verses anchor each cluster. The inflection point is completion for the full registry of an OWNER term — every verse classified, every group anchored, every word re-exported in its updated form.

**Dimension Review — Dimensional placement.** The groups produced by Verse Context are then read across a cluster of related registries at once, and each group is assigned a dimension of the inner being. The dimensions are data-derived — a working vocabulary that emerged from the corpus and that is refined by continued contact with it, not a list imposed from theological tradition or psychological category. A dimension that cannot be grounded in at least one verse in the registry's corpus is not a dimension for that registry. The Dimension Review also validates that each group's description remains characteristic-centric and corrects groups that had been formed under an earlier, term-centric model. Its output is a set of confirmed group placements, ready for analytical work. Its inflection point is the cluster's completion: every active registry in the cluster with every group dimensioned and locked.

**Session B — Word analysis.** With verse context grouped and dimensioned, Session B runs the deep analytical pass on each word in turn. Session B is organised in two halves — a readiness half that establishes the state of the data and the analytical frame for the word, and an output half that produces the written work. A biblical pass is done first: semantic characterisation of the word, dimensional placement, treatment of the divine and somatic dimensions as they appear in the verses, relational and causative patterns, anchor-verse analysis. A scientific pass follows — never simultaneous — engaging the relevant contemporary research at defined points. Every finding is substantiated by data in the database, and every answer is persisted there. The analytical process is contained by this persistence: on re-reading, prior findings are set aside and the data is read freshly. Session B's output is the complete analytical record for the word. Its inflection point is a word whose analysis is closed — every standing question answered, every finding traceable to evidence, the record ready for publication.

**Session C — Publication.** Session C produces the reader-facing written study for each word. It is the primary presentation document. It reads from Session B's analytical record and presents the word in a form a reader can work with — structured, navigable, written for the audience described in the expected-outcome sub-section. Session C stands on its own: a reader encountering a single word study gains a considered account of that aspect of the inner being, grounded in the evidence and transparent about what remains uncertain. Session C is produced one word at a time, and a word's publication depends on its analysis being closed.

**Session D — Cross-registry synthesis.** Once every word in a cluster has moved through Session C, Session D reads the results across the cluster as a whole. It produces the second-order findings that are not visible at the single-word level: where two words turn out to be different angles on a single reality, where one word splits into two that had been wrongly conflated, where a characteristic runs across a group of words in a consistent way. Session D is the instrument of synthesis at cluster scale; beyond it lies the programme-level synthesis that becomes possible only when all clusters are complete. Session D's inflection point is the cluster-level account: a written synthesis that adds to the per-word studies what they cannot say alone.

The pipeline handles a word once at each phase and carries its state between phases through the database. When a word is at Verse Context, the database records that state; when it moves to Dimension Review, the database is updated; Session B reads the state to know which words are ready to receive analytical work. The state is also what allows the programme to pause and resume safely — a session is a span of work on words that are ready for that phase, not a fixed timetable, and work can be taken up at whatever point the database shows it left off.
