### Three within-sub-group polysemy tasks (MANDATORY — flagged by Phase 6)

These were deliberately left for Phase 7 to handle via VCG-level distinction. Each requires a **separate VCG** within its sub-group:

1. **M10b-A — ra.sha forensic-verdict sub-corpus** (~5V: Exo 23:7, Deu 25:1, 25:2, 1Ki 8:32, 2Ch 6:23). M10 cross-register flag. These verses use ra.sha as a *legal status label* (the wicked = the legally condemnable) rather than naming inner moral character. Design a distinct VCG (e.g. `M10b-A-VCG-{N}` "forensic-verdict register"). The remaining ~185 ra.sha verses cluster by inner-moral-character themes.

2. **M10b-B — ponēros cosmic-evil-agent sub-corpus** (~10–15V: Mat 13:19, Mat 13:38, Joh 17:15, Eph 6:16, 2Th 3:3, 1Jn 2:13, 1Jn 2:14, 1Jn 5:18, 1Jn 5:19, Gal 1:4, Eph 5:16, Eph 6:13). M27 cross-register flag. These verses name "the evil one" / "the evil age" — cosmic/supra-personal evil rather than personal-character evil. Design a distinct VCG (e.g. `M10b-B-VCG-{N}` "cosmic-evil-agent / evil-one / evil-age"). The remaining ~50 ponēros verses cluster by personal-character-evil themes.

3. **M10b-E — a.ven trouble/distress sub-corpus** (~18V: the affliction-suffered uses across Psa 55:3, 55:10, 88:8?, Pro 22:8, Job 5:6, Job 31:3 — read the meanings to identify the full set). M03/M27 cross-register flags. These verses name *the harm/trouble that befalls the victim* rather than the iniquity-scheming of the agent. Design a distinct VCG (e.g. `M10b-E-VCG-{N}` "trouble/affliction-suffered register"). The remaining ~48 a.ven verses + 13 ro.a verses cluster by the iniquity-scheming / evil-deeds themes.

## Process per sub-group (§10.2 + §10.7 staged write-out)

1. **Read** every verse-meaning + keyword set in the sub-group's section of the meanings report. **Every row. No skipping. No sampling.**
2. **Cluster meanings into provisional VCGs.** Use the per-sub-group keyword breakdowns in the analytics report (§6) as a starting point — high-frequency keyword patterns within a sub-group are candidates for VCG groupings. Verify against the meanings.
3. **Name each VCG** with a provisional code (`M10b-{X}-VCG-{seq}`, e.g. `M10b-A-VCG-01`) and a one-paragraph `context_description`.
4. **Designate ONE anchor verse per VCG** — most directly and definitionally evidences the phenomenon. Anchor's vc_id must be in the VCG's member list.
5. **Note dual-membership verses** explicitly (rare; usually within the same sub-group).
6. **Write the per-sub-group design document to disk immediately:**
   `Sessions/Session_Clusters/M10b/wa-cluster-M10b-{subgroup_code}-vcg-design-v1-20260525.md`
   (e.g. `wa-cluster-M10b-M10b-A-vcg-design-v1-20260525.md`)
7. **Verify sum.** Sub-group's VCG member sum must equal the sub-group's verse count from the meanings report. Record a verification line at the end of the per-sub-group document:
   ```
   **Verification**: VCG member sums = N1 + N2 + ... + Nk = TOTAL, matches M10b-{X} input count of TOTAL ✓
   ```
8. Move to next sub-group; repeat.

After all 6 sub-groups are processed, produce:

- **Unified VCG creation JSON**: `Sessions/Session_Clusters/M10b/wa-cluster-M10b-vcg-creation-v1-20260525.json`. Top-level keys are sub-group codes; each contains a `vcgs` array. Every VCG has `provisional_code`, `description`, `verses` (complete array, every vc_id), `anchor_vc_id` (must be member of `verses`).
- **Cross-routing flags document** if any verses surface as needing routing review: `Sessions/Session_Clusters/M10b/wa-cluster-M10b-phase7-cross-routing-flags-v1-20260525.md` (probably empty for M10b — flag-distribution has already been resolved at Phase 5/6).
- **Pre-submission checklist** verification (§10.8 — confirm every item before declaring complete).
