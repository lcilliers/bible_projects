## Process per sub-group (§10.2 + §10.7 staged write-out)

1. **Read** every verse-meaning + keyword set in the sub-group's section of the meanings report. **Every row. No skipping. No sampling.**
2. **Cluster meanings into provisional VCGs.** Use the per-sub-group keyword breakdowns in the analytics report (§6) as a starting point — high-frequency keyword patterns within a sub-group are candidates for VCG groupings. Verify against the meanings.
3. **Name each VCG** with a provisional code (`M10-{X}-VCG-{seq}`, e.g. `M10-J-VCG-01`) and a one-paragraph `context_description`.
4. **Designate ONE anchor verse per VCG** — most directly and definitionally evidences the phenomenon. Anchor's vc_id must be in the VCG's member list.
5. **Note dual-membership verses** explicitly (rare; usually within the same sub-group).
6. **Write the per-sub-group design document to disk immediately:**
   `Sessions/Session_Clusters/M10/wa-cluster-M10-{subgroup_code}-vcg-design-v1-20260523.md`
7. **Verify sum.** Members must equal input count.
8. Move to next sub-group; repeat.

**M10-E (Iniquity as accumulated crime, 162V) — single term.** All 162 verses are a.von H5771G. VCGs partition by kind-of-iniquity (generational, communal, heart-seated, accumulated, hidden, exposed). Already noted in v1 brief.

Read for sub-distinctions: M10-G splits by covenant-betrayal-against-God vs marital/relational; M10-J by sin-against-God vs sin-against-neighbour vs deliberate-defiance-after-warning; M10-D by guilt-recognition-process vs guilt-as-weight-borne vs guilt-removal.

## Discipline (§10.7 + §10.8)

1. **Read every verse-meaning + keyword set** for each sub-group. No sampling.
2. **Stage by sub-group.** After each sub-group: write design doc, verify sum, move on. Don't try to hold multiple sub-groups in working memory.
3. **Use the keyword analytics within each sub-group** to identify VCG candidates. The §6 per-sub-group breakdowns show which keywords concentrate where — that's evidence of VCG-level groupings.
4. **Sum verification per sub-group.** Members must equal input count.
5. **Anchor in members.** Every `anchor_vc_id` must be in its VCG's `verses` array.
6. **No phantom vc_ids.** Every vc_id used must be in the meanings report.
7. **No vc_id in two VCGs** unless explicitly dual-membership flagged (rare).
8. **M10-BND single VCG** holding all 5 BOUNDARY verses — do NOT subdivide.
9. **Preserve cross-register flags** in VCG descriptions per the catalogue above.
