As of 2026-05-23, Phase 2 Pass A produces **two outputs per verse in a single API call**: the one-line meaning (`verse_context.analysis_note`) AND a set of 3–7 atomic inner-being keywords (`verse_context.keywords`, JSON array). Schema bumped to 3.25.0 via migration M51.

**Why this matters.** The keywords are atomic and compositional (1–2 word tokens; no particles, no proper names, no sentences — see [[feedback-keyword-discovery-interim-phase]] for the format discipline). Their value is in aggregation: the same `conscience` or `will refusing` or `defection wilful` appears across many verses, so a cluster's keyword pool reveals which inner-being faculty axes are present in the corpus — independent of the gloss-level term names. The pool is the structural input that drives Phase 5 sub-group design.

**How to apply.**

1. **Phase 2 Pass A** (`scripts/_run_passa_via_api_v1_20260515.py --m-cluster {CODE}`) — produces both fields. The VCREVISE patch's `set` block carries `{"analysis_note": "...", "keywords": "[\"kw1\", \"kw2\", ...]"}`. The patch applier passes both fields through.
2. **Pre-Phase 5**, CC generates the cluster-level keyword analytics: `scripts/_generate_cluster_keyword_analytics_v1_20260523.py --cluster {CODE}`. Output: `Sessions/Session_Clusters/{CODE}/wa-cluster-{CODE}-keyword-analytics-v{N}-{date}.md`. Sections: top-50 keywords, top-30 tokens (faculty-axis indicators) with sample families, top-40 co-occurrence pairs, per-term keyword density, per-sub-group breakdown (if sub-groups exist).
3. **Phase 5 brief** lists the keyword analytics report as a structural input alongside the constitution report. The brief instructs the AI to use the keyword pool to identify faculty axes BEFORE assigning verses to sub-groups — keywords surface the axes; meanings ground them in evidence.

**Backfill for legacy clusters.** For closed clusters that predate the embedding, the per-sub-group discovery script (`_keyword_discovery_subgroup_v1_20260523.py`) and DB loader (`_load_keywords_to_db_v1_20260523.py`) backfill `verse_context.keywords` from external JSON files. Backfill is invoked only on a need-to-do basis (e.g. when a closed cluster is reopened for Phase 5 revision) — see [[feedback-no-rework-paid-twice]] (don't retroactively backfill the whole programme).

**M10 precedent.** First cluster to use the embedded flow + backfill. M10 was in Phase 5 revision when the embedding was applied: M10-A/B/F had keywords from the discovery script (backfilled into DB); M10-C/D/E/G/H/I/BND had keywords generated by the discovery script (also backfilled) — total 1,325 keyword-tagged verses. Cluster-level analytics produced and fed into the Phase 5 revision brief for M10-A and M10-B.
