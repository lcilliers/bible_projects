When a directive applies the first analytical operation to a cluster currently at `cluster.status = 'Not started'` (e.g. Phase 2 UT-review patch, Phase 4 sub-group assignment, etc.), the apply must also update `cluster.status` to `'Analysis - In Progress'` in the same commit.

**Why:** Programme convention established by M05 / M06 / M32 / M15 work. The status field is what scripts and reports use to decide which clusters are active. Leaving a cluster at 'Not started' after analytical work has begun creates a misleading state — `_generate_cluster_overview_v1_*` will show 'Not started' even though sub-groups exist or vc rows have been authored.

**Why specifically caught here:** On 2026-05-08 the M15 Phase 2 UT-review canonical patch was applied and committed without advancing status; user had to flag the omission. The directive itself didn't include a status update because the patch envelope only carries verse_context operations — the directive author left status as a separate concern.

**How to apply:**
1. Before any cluster-process directive write, check current `cluster.status`.
2. If `'Not started'` and the directive produces an analytical write (any of: cluster_subgroup INSERT, mti_terms.cluster_subgroup_id assignment, verse_context INSERT/UPDATE on cluster's terms, cluster_finding INSERT), advance to `'Analysis - In Progress'` in the same transaction or as an immediately following commit.
3. The advance is independent of the directive's explicit instructions — it's programme convention.
4. When a cluster's analytical work completes (Phase 9 findings recording done, researcher confirms), advance to `'Analysis Completed'` (M05 / M06 precedent).

**Refresh both cluster-overview reports after any cluster-state change:**

There are two cluster-overview series with different scope/format:
- `Workflow/Clusters/wa-cluster-overview-vN-YYYYMMDD.md` (+ json sibling) — richer canonical version with bucket / coverage / per-term-status columns + detail-file links. Generated by `scripts/_generate_cluster_term_report_v1_20260505.py` (auto-bumps vN). Also regenerates 46 per-cluster `wa-cluster-<code>-detail-v{N}-{date}.md` files. **This is the version that pairs with each session's commits** and was missed on 2026-05-08 until the user flagged it.
- `Workflow/Programme/Program_reports/wa-cluster-overview-{date}.md` — simpler programme-level summary. Generated by `scripts/_generate_cluster_overview_v1_20260508.py`. Idempotent per day (overwrites on rerun).

After any cluster_code, cluster_subgroup, or cluster_finding write, refresh **both**.
