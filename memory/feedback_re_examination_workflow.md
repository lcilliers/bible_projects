Closed clusters are not immutable. Four categories of post-closure work require re-opening a cluster under a structured workflow:

1. **Unresolved BOUNDARY backlog** — verses parked under the pre-correction methodology (M01: 7, M02: 4, M03: 28; M04 about to incur ~322 unless v2_5 is applied first).
2. **New verses added** — STEP re-extraction or registry additions surface verses not in the cluster at original analysis time.
3. **Verses skipped or mis-set-aside** — discovery, during a later cluster's analysis, that a verse was wrongly removed.
4. **Scope-correction re-classification** — application of [[feedback-inner-being-full-scope]] to verses that were set aside under the old "no spiritual link" filter.

**Why:** The user observed that completed clusters M01, M02, M03 (and M15, pending audit) carry artefacts of the pre-correction methodology. Treating closure as final would leave those clusters carrying defective analytics indefinitely. The user explicitly framed this as "the start of the thinking process" — meaning the workflow needs design before it can be applied.

**How to apply:**
- New §16 (Re-examination Workflow) in the cluster instruction. Required elements:
  - Re-open trigger: explicit researcher decision recording the trigger category (BOUNDARY backlog / new verses / mis-set-aside / scope re-classification).
  - Cluster status transitions back from "Closed" to "Re-examination - In Progress" (new status value).
  - Bounded scope: re-examination touches only the affected verses, not the full cluster corpus — existing sub-group assignments, VCGs, and findings remain unless the re-examined verses force a structural change.
  - Output: a delta patch (verses added / verses set aside / verses re-routed / new sub-group entries / new VCG entries / findings amendments).
  - Closure re-attestation: after delta applied, cluster re-closes with audit trail showing the trigger and delta.
- Apply order for the M01/M02/M03 backlog: smallest first (M02 → M01 → M03 → M15 audit → then onward), to build confidence in the workflow before tackling the larger M03 BOUNDARY count.
